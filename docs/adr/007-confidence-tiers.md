# ADR-007: Evidence-Based Confidence Tier Assignment

**Status:** Accepted
**Date:** 2026-03-10
**Deciders:** Pipeline engineering, knowledge graph team
**Relates to:** ADR-005 (Provenance), ADR-004 (GraphGuard)

---

## Context

The v0.2.0 pipeline committed enrichments with no confidence signal. Every committed
attribute had equal epistemic weight, regardless of whether it was backed by three
authoritative sources or generated by the LLM with zero external grounding.

This created a trust problem: downstream consumers (analysts, automated systems) had
no way to distinguish:
- A vendor risk tier confirmed by regulatory filings (high confidence)
- A system owner inferred by the LLM from job posting patterns (medium confidence)
- A person's role speculated by the LLM from their name alone (low confidence)

The hc-enterprise-kg platform already uses a T1–T4 confidence tier system for
the CDAIO dataset. Making the enrichment pipeline produce the same tiers ensures
consistency across the full platform ecosystem.

---

## Decision

Introduce `ConfidenceAgent` as stage 4 in the 7-agent pipeline, running after
`ReasoningAgent` and before `CoherenceAgent`.

### Agent Role

`ConfidenceAgent` is deterministic and rule-based (no LLM call). It evaluates
signals from the payload and assigns a T1–T4 tier and a 0.50–1.00 score.

### Scoring Rubric

| Signal | Maximum Contribution |
|--------|---------------------|
| Source count (≥3 sources) | +0.20 |
| Source quality (avg relevance ≥0.8) | +0.08 |
| Entity name in search queries | +0.05 |
| Grounding language in reasoning | +0.06 |
| Specific (non-generic) attribute values | +0.05 |
| Graph-corroborated relationships | +0.03 |
| Hedging language penalty | −0.10 |
| Generic attribute value penalty | −0.05 |
| **Baseline (LLM judgment alone)** | **0.50** |

### Tier Boundaries

| Tier | Score | Semantic Meaning |
|------|-------|-----------------|
| T1 | 0.94–1.00 | Verified Fact — multiple authoritative sources |
| T2 | 0.80–0.93 | Strong Inference — multiple corroborating sources |
| T3 | 0.65–0.79 | Reasoned Inference — limited/indirect evidence |
| T4 | 0.50–0.64 | Working Hypothesis — LLM judgment, minimal grounding |

### Pipeline Integration

The `confidence_tier` and `confidence_score` are injected into the `AgentMessage.payload`
after `ConfidenceAgent.run()` and read by:

1. **CommitAgent** — written to entity provenance and relationship provenance
2. **AuditLog** — recorded in every `AuditEvent`
3. **EnrichmentRun** — visible in run-level stats (via tier distribution metrics)
4. **EnrichmentMetrics** — tracked in `confidence_tier_total` counter family

---

## Why Rule-Based, Not LLM-Based

An LLM-based confidence evaluator would:
- Add a second LLM call per entity (cost and latency)
- Introduce the very uncertainty it is trying to measure
- Produce inconsistent results across similar evidence sets

The rule-based approach is:
- Deterministic: same evidence = same score, always
- Fast: no network call, sub-millisecond execution
- Testable: every rubric element is independently verifiable
- Explainable: `confidence_rationale` field shows exact score breakdown

---

## Consequences

**Positive:**
- Every committed enrichment carries a machine-readable epistemic signal
- Downstream systems can filter on `confidence_tier` (e.g., only use T1/T2 for compliance reporting)
- Hedging language detection provides an automatic quality signal for LLM outputs
- Consistent with hc-enterprise-kg platform tier system

**Negative:**
- Scoring rubric requires calibration as the platform matures
- T4 enrichments will still be committed by default — operators may want to configure a minimum tier
- Source relevance scores from Tavily are used directly; different search providers may calibrate differently

**Rejected alternatives:**
- **LLM-based confidence evaluation:** Too expensive; circular reasoning
- **Simple source count threshold:** Too coarse; ignores reasoning quality signals
- **No confidence tracking:** Unacceptable for enterprise compliance use cases

---

## Re-evaluation triggers

- Analysis shows T4 enrichments are frequently incorrect → add configurable `min_confidence_tier` gate
- Rubric calibration data available from user feedback → update scoring weights
- New search providers added → validate relevance score semantics
